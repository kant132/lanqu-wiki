# Phase 5 风险建模与业务安全评估

## 5.2 五维风险暴露权重算法

### a) 一票否决

若参数校验发现 Local_Sanitizer == true（强正则/白名单），直接将该端点标记为 priority: LOW，score=1，熔断不追踪。

### b) 评分计算

```
Risk_Score = 鉴权因子 × 参数校验因子 × 高危参数 × 业务意义
```

> **乘法模型说明**：采用乘法而非加法，是因为纵深防御中任何一层的强防护都能显著降低整体风险。但需注意：若 Phase 2/3 已发现 Filter/Interceptor 绕过漏洞，则该端点的 `全局过滤因子` 必须强制取 5（无任何全局过滤拦截），不得因代码中存在 Filter 声明而给予低分。

### c) 因子分值表

统一：LOW=1，MEDIUM=3，HIGH/CRITICAL=5

| 因子 | LOW (1分) | MEDIUM (3分) | HIGH / CRITICAL (5分) |
|------|-----------|--------------|----------------------|
| 鉴权因子 (Auth) | AdminOnly（高权限角色） | Authenticated（普通登录凭证） | Anonymous / PermitAll（匿名可访问） |
| 全局过滤因子 (GlobalFilter) | 强全局规范化与净化器 | 弱全局过滤器（仅审计/日志） | 无任何全局过滤拦截 |
| 参数校验因子 (Validation) | 强正则 / 白名单 / 本地净化 | 基础类型 / 长度限制校验 | 零校验（裸奔接口） / 弱校验 |
| 高危参数因子 (DangerousParam) | 无敏感参数 | 有限敏感参数（用户名/ID/枚举） | 高危参数（命令执行、文件操作、SQL注入、反序列化、SSRF/HTTP请求转发、重定向、XXE、LDAP/SpEL、路径穿越） |
| 业务意义因子 (BusinessImpact) | 测试接口 / 演示接口 / 已废弃接口 | 普通业务接口（查询、列表、非敏感更新） | 核心业务接口（交易、支付、权限变更、敏感数据读写、管理操作、认证、添加资源等） |

### d) 风险等级重定义与动作

| 风险等级 | 分值范围 | 动作 |
|----------|----------|------|
| CRITICAL | ≥60 | 必选启动LSP深度追踪，并分析具体业务对应的安全风险 |
| HIGH | 30-59 | 必选启动LSP深度追踪，并分析具体业务对应的安全风险 |
| MEDIUM | 10-29 | 可由主Agent根据剩余算力令牌选择性追踪 |
| LOW | ≤9 | 强制熔断，输出算力裁剪日志 |

## 5.3 基于代码分析的动态业务安全评估

Phase 5 的业务安全评估应由AI根据代码内容自主分析，而非依赖预设关键词。

### 5.3.1 分析流程

| 步骤 | 输入 | 分析操作 | 输出 |
|------|------|----------|------|
| 1 | API方法名、参数签名 | 分析方法签名和参数含义 | 推断API用途和操作对象 |
| 2 | Controller/Service/DAO代码 | 分析调用链，理解方法职责 | 关联业务场景 |
| 3 | 代码中安全相关API | 识别加密算法、认证机制、授权检查 | 发现安全技术 |
| 4 | 业务逻辑和参数可控性 | 推导潜在风险类型 | 可能存在的漏洞 |
| 5 | 潜在风险对应的代码 | LSP追踪验证漏洞是否存在 | 确认漏洞及代码证据 |

### 5.3.2 AI自主分析要点

**第一步：理解业务上下文**

- 分析方法签名：`downloadBackup(String fileId)` → 文件操作相关
- 分析参数含义：`userId`, `fileId`, `accountNo` → 资源标识符
- 分析方法调用链：从Controller到Service到DAO，理解完整业务流
- 推断业务场景：结合类名、方法名、参数名推断实际用途

**第二步：识别安全技术**

- 加密算法：检查是否使用 `AES`, `RSA`, `DES`, `BCrypt`, `SHA` 等
- 认证机制：检查 `@PreAuthorize`, `hasRole()`, `isAuthenticated()` 等
- 授权检查：检查用户ID与操作对象的归属关系
- 会话管理：检查 `HttpSession`, `JSESSIONID`, `Token` 处理
- 输入处理：检查 `@Valid`, `@Pattern`, `Sanitizer` 等

**第三步：推导潜在风险**

- 业务逻辑漏洞：用户可操作他人的资源（IDOR）
- 加密实现漏洞：硬编码密钥、弱加密模式、不安全的IV
- 认证绕过漏洞：缺失鉴权注解、鉴权逻辑可绕过
- 注入漏洞：SQL注入、XSS、命令注入、路径穿越

**第四步：代码验证**

- 对推导出的每个潜在风险，检查代码是否确实存在该问题
- 记录验证结果（有漏洞/无漏洞）和证据（代码位置）
