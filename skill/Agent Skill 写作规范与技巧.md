## 一、认知升级：为什么是「技能」而不是「提示词」？

### 1.1 传统 Prompt 的困境

传统 Prompt Engineering 的基本假设是：把需求告诉模型，模型就能理解并执行。但在复杂场景下，这一假设不断崩塌：

- 模型一边要理解任务，一边要判断该调用什么工具
- 大段提示词里指令相互冲突，注意力被稀释
- 路由极不稳定，同一请求可能触发完全不同的执行路径

### 1.2 Skill 的范式转变

Agent Skills 给出的答案：**放弃"凡事都说清楚"的思路，换一种全新的范式**

> **Prompt 是"一次性指令"，Skill 是"可重复使用的能力包"**

Skill 不是一段提示词，而是一个完整的**上下文封装（Context Packaging）**，目标不是给人阅读，而是「给模型提供正确的运行时上下文」。

### 1.3 核心认知

```
Prompt Engineering → 告诉模型"说什么"
Agent Skills → 给模型"正确的运行时上下文"
```

## 二、工程化底座：Skill 的标准结构

### 2.1 文件夹结构

一个标准的 Skill 是一个组织严密的文件夹：

```
my_skill/
├── SKILL.md           # 核心指令（YAML frontmatter + 主逻辑）
├── scripts/           # Agent 可直接运行的代码
├── references/        # 按需加载的参考文档
├── assets/            # 模板、Schema、静态资源
├── examples/          # 示例输入输出
└── config.json        # 首次运行的用户配置
```

**设计动机**：模型在多选一的精度，跟选项数量呈负相关。把太多内容挤在一个文件里，Agent 反而不知道该看哪里。

### 2.2 SKILL.md 核心模板

```markdown
---
name: skill-name                    # 全小写、无空格、可用连字符
description: Load when user asks... # 可路由描述，不是功能说明
---

# 技能名称

## 触发条件
当用户...时应调用此技能

## 执行步骤
1. 步骤一：...
2. 步骤二：...

## 输出要求
- 输出格式：...
- 质量标准：...

## 约束与限制
- 仅限于...
- 禁止...
```

### 2.3 YAML Frontmatter 核心字段

| 字段 | 要求 | 错误示例 | 正确示例 |
|------|------|---------|---------|
| `name` | 必须与目录名完全一致 | `My Skill` | `code-review` |
| `description` | 写成"Load when..."而非"This Skill does..." | `执行代码审查` | `Load when user asks to review code security` |

## 三、上下文经济学：Perplexity 的「三层税制」

Perplexity 团队提出：**Context 是有定价的，每个 Token 都要被征收"税"**。

### 3.1 三层税制结构

| 层级 | 加载内容 | 预算(Token) | 何时支付 | 核心功能 |
|------|----------|-------------|---------|---------|
| **第一层：Index** | 所有可见 Skill 的 name + description | ~100 tokens/个 | 每个会话、每个用户 | Skill Routing |
| **第二层：SKILL.md** | 完整正文 | ~5000 tokens | 加载后到上下文压缩边界 | 核心指令与工作流 |
| **第三层：Heavy Assets** | references/、examples/、assets/ | 动态 | 按需读取时 | 深度参考信息 |

### 3.2 优化原则

- **Index 税是避不掉的**——description 写精准，减少误匹配
- **第二层税决定整体效率**——SKILL.md 内容质量是关键
- **第三层税按需加载**——不 premature load，避免浪费

### 3.3 一句穿透式判断标准

> **"模型没看到这句会出错吗？"** 通不过就删。

这条标准颠覆了传统的文档写作习惯：

| 应该删掉 | 应该保留 |
|---------|---------|
| 模型本来就会的基础操作 | 模型不知道的"坑" |
| "用 Git 提交代码" | "不要修改 migration 文件" |
| "创建 Python 文件" | "这个 API 有 rate limit，必须先查剩余配额" |
| 好解释的内容 | 必须解释才能理解的内容 |

### 3.4 生产级的「树状分层」案例

**反面教材**：把 1945 条 IRS 税法全塞进一个文件夹。

结果：Agent 表现比不加载 Skill 还差——在巨型平铺列表里找不到正确条目，开始产生幻觉。

**解决方案**：重构为三层 Taxonomy
```
300 topics → 20 areas → ~15 topics 以内
```

配合自定义搜索工具，才把任务能力做扎实。

**核心工程原则**：**分层检索本身就是 Skill 的 Feature，不是 Overhead。**

## 四、五大设计模式：Google 总结的工程「决策库」

### 模式一：工具封装器（Tool Wrapper）

**核心逻辑**：将特定技术的规范封装为按需加载的知识包，代替冗长的 System Prompt。

```markdown
## 加载规则
当用户提到 "FastAPI"、"REST API"、"Pydantic" 时，
动态加载 references/ 中的规范文档
```

**优势**：
- 上下文窗口干净
- 规范更新只需改 Skill 文件
- 团队规范可打包分发

### 模式二：生成器（Generator）

**核心逻辑**：用模板 + 风格指南强制输出一致性。

Generator 模式像项目经理：
1. 加载模板
2. 加载风格指南
3. 向用户询问缺失变量
4. 按模板结构填充

**解决痛点**：每次输出的文档结构都不一致

### 模式三：审查器（Reviewer）

**核心逻辑**：将"检查什么"与"如何检查"分离。

```
用户提交代码 → Agent 加载 checklist → 分项评分 → 按严重程度分组结果
```

将 Python 风格检查清单替换为 OWASP 安全检查清单，同一个 Skill 基础设施可以执行完全不同的专业审计。

### 模式四：反转（Inversion）

**核心逻辑**：Agent 在执行前先"访谈"用户。

```
Agent 不直接行动 → 先向用户提出澄清性问题 → 确保方向正确后继续执行
```

**核心原则**：程序性知识不只是"怎么做"，还包括"什么时候需要问"。

### 模式五：流水线（Pipeline）

**核心逻辑**：强制执行带有检查点的严格多步骤工作流。

```
阶段1 → 验证 → 阶段2 → 验证 → 阶段3 → 输出
```

适用于：
- 文档生成流水线
- API 文档自动生成
- 项目脚手架搭建

### 模式组合使用

五种模式可以组合使用，构成 Agent Skill 的「设计决策库」。

## 五、五种「Skill 之禅」vs「Python 之禅」

Perplexity 团队整理的对照表：

| Python 之禅 | Skill 之禅 |
|------------|-----------|
| Simple is better than complex | Skill 是一个文件夹，复杂性本身就是 Feature |
| Explicit is better than implicit | 激活靠隐式模式匹配，靠渐进披露 |
| Sparse is better than dense | Context 很贵，每个 Token 都要带最大信号 |
| 实现好解释就是好主意 | 如果好解释，说明模型已经知道——删掉 |

> **写代码是给人读的，写 Skill 是给模型用的。这是两种完全不同的语境。**

## 六、验证的科学：OpenAI 的 Eval 体系

### 6.1 四条失效路径

| 失效类型 | 具体表现 | 最致命之处 |
|---------|---------|-----------|
| **根本未触发** | 用户要求格式化代码，Agent 用自己的知识乱改 | 最隐蔽，手测几乎发现不了 |
| **触发了但未完成** | 应该创建 3 个文件只建了 2 个；迁移中途退出 | 影响最大，用户看到的就是结果 |
| **结果对但过程错** | 备份 → 迁移 → 验证，顺序错成先迁移再备份 | 纯结果验证完全看不到 |
| **完成但质量不达标** | 代码风格不一致；Commit message 格式错；Token 浪费 | 在团队协作和成本控制上积累问题 |

### 6.2 四维验证目标

| 维度 | 问题 | 验证什么 |
|------|------|---------|
| **Outcome（结果）** | 任务完成了吗？应用能正常运行吗？ | 最终产出 |
| **Process（过程）** | Codex 是否调用了指定技能并按预期顺序执行了工具？ | 执行路径 |
| **Style（风格）** | 输出是否符合要求的规范？ | 符合度 |
| **Efficiency（效率）** | 是否避免了冗余操作或过度消耗 Token？ | 成本控制 |

### 6.3 评分规则型 Eval

不是简单通过/失败，而是建立可追踪、可对比的评分体系，让 Agent 的行为变化可量化、可回溯。

## 七、评估驱动优化（Eval-Driven Engineering）

### 7.1 新版 Skill-creator 工作流

```
草稿生成 → 评估 → 迭代优化
    ↑                      ↓
    ←←←←←←←←←←←←←←←←←←←←←
```

### 7.2 触发评估指标

| 指标 | 公式 | 目标 |
|------|------|------|
| Precision | TP/(TP+FP) | 触发准确率 |
| Recall | TP/(TP+FN) | 召回率 |
| Accuracy | (TP+TN)/Total | 总体准确率 |

### 7.3 Subagent 分工

| 角色 | 职责 | 输出 |
|------|------|------|
| Executor | 并行执行任务 | 执行记录 |
| Grader | 验证断言、批判测试用例 | grading.json |
| Comparator | 盲测 A/B 对比 | comparison.json |
| Analyzer | 归因分析、生成建议 | analysis.json |

### 7.4 Grader Agent 评判标准

**PASS 条件**：
- 有明确证据支持断言
- 证据反映实质性内容（非表面合规）
- 任务结果正确且完整

**FAIL 条件**：
- 无证据支持
- 证据与断言矛盾
- 断言被「巧合满足」而非真正完成

**额外检查**：
- 事实性声明（factual claims）
- 流程性声明（process claims）
- 质量声明（quality claims）

## 八、与 MCP 的本质区别

| 维度 | Agent Skills | MCP 架构 |
|------|-------------|----------|
| **颗粒度** | 原子级技能单元 | 粗粒度服务组合 |
| **执行模式** | 同步闭环处理 | 异步事件驱动 |
| **状态管理** | 内置上下文记忆 | 依赖外部状态存储 |
| **适用场景** | 复杂确定性流程 | 简单松散耦合任务 |

**关系**：Skills 承载「怎么做」的程序性知识，MCP 承载「用什么工具」的声明性接口。**两者是互补关系，不是替代关系。**

## 九、Skill 演进路径：从 Skill 到 Agent 智能体

### Level 0：原始 Prompt
每次手工输入指令，过程不可复用。

### Level 1：Task-Prompt Skill
打包一个 Markdown 指令文件，按需加载。

### Level 2：Resource-Bundled Skill
加入 references/、assets/ 和 scripts/，支持更复杂任务。

### Level 3：Workflow Skill
整合 Generator、Reviewer 和 Inversion 模式，支持多步骤流程和用户交互。

### Level 4：Multi-Agent Orchestration
Skills 之间相互调用和协同，实现跨领域的 Agent 工作流。

## 十、最佳实践清单

### 10.1 触发描述（Description）
- [ ] 使用具体场景而非抽象概念
- [ ] 区分相似技能的边界
- [ ] 包含负面示例（当...而非...时）
- [ ] 添加限制条件
- [ ] 写成"Load when..."而非"This Skill does..."

### 10.2 指令编写（SKILL.md）
- [ ] 每条指令回答「没有会怎样？」
- [ ] 步骤分解到可执行粒度
- [ ] 指定具体工具名称
- [ ] 说明边界条件和例外
- [ ] 提供典型示例
- [ ] 定义清晰输出格式
- [ ] 删除"好解释"的内容——说明模型已经知道

### 10.3 结构组织
- [ ] 使用文件夹结构，而非单文件
- [ ] 大型参考使用分层检索
- [ ] 按需加载，而非全部加载

### 10.4 测试验证
- [ ] 编写功能测试用例（覆盖正常路径）
- [ ] 编写触发测试用例（区分应触发/不应触发）
- [ ] 覆盖异常路径和边界条件
- [ ] 四维验证：Outcome、Process、Style、Efficiency

## 十一、常见反模式

| 反模式 | 问题 | 修正方案 |
|--------|------|---------|
| 指令过于抽象 | Agent 无法执行 | 分解为具体步骤 |
| 缺少边界条件 | 意外行为 | 明确限制和例外 |
| 冗余信息 | 上下文浪费 | 删除"显而易见"的内容 |
| 单一评分 | 可被 Hacking | 多维度 Rubric |
| 无测试覆盖 | 质量不可信 | 完整评估体系 |
| 巨型平铺列表 | 幻觉率上升 | 分层检索 + 搜索工具 |
| 解释性内容过多 | Token 浪费 | 只保留"模型不知道的坑" |

## 十二、2026 年 Skill 生态全景

| 平台 | Skill 支持情况 |
|------|--------------|
| **Anthropic Claude Code** | 内置 slash-command 访问 Skills，支持并行 MCP 连接、Prompt Cache 控制 |

### 两种 Skill 类型

1. **能力提升型 Skill**：教模型做基础能力之外的事（如文档创建、编码技巧集成）
2. **流程结构化 Skill**：用于多 Agent 加速执行、触发优化和减少误判

> 2026 年 AI 的竞争，正在从「拼模型」转向「拼 Skill 生态」。
