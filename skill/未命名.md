# Agent Skill 写作完全指南：步骤 · 技巧 · 规范 · 注意事项

> 基于 Anthropic、Google ADK、OpenAI、Perplexity 2026 年最新实践的系统性总结


## 一、Skill 写作的六步法

### Step 1：判定「是否值得写成 Skill」

**先问三个问题：**

1. 这个任务模型凭自己的知识会做错吗？
2. 这个任务在团队/项目中会重复发生吗？
3. 这个任务的正确做法有「坑」需要记录吗？

**三个「否」→ 不写 Skill。** 写一个 Skill 是有成本的——索引税、维护税、认知税。不是为了复用而写，是为了「防止模型犯错」而写。

> **Perplexity 的一句决断标准**：「模型没看到这句会出错吗？」通不过，就删掉整个 Skill。

---

### Step 2：定义触发条件（Description 写作）

这是 Skill 最容易被搞砸的一步。

**错误写法**（描述 Skill 本身）：
```
This skill helps with React component development and UI styling
```

**正确写法**（描述「什么情况下加载」）：
```
Load when user asks to build a new React component, mentions UI styling with Tailwind, or needs component architecture guidance
```

**黄金公式**：
```
Load when [用户的关键词/动作] + [场景限定]
```

Description 直接决定了 Agent 的 **Index 税**能否被有效「缴出去」——描述写得准，Index 阶段的 100 tokens 就花得值；写不准，后续全白费。

---

### Step 3：构建 SKILL.md 核心内容

SKILL.md 的结构有严格的优先级顺序：

```markdown
---
name: skill-name（全小写，与目录名一致）
description: Load when [触发条件]
---

# Skill 名称

## 核心规则（必须遵守）
- 规则1
- 规则2

## 工作流程
1. 第一步
2. 第二步

## 常见错误
- 错误场景 → 正确做法

## 参考（指向 references/ 中的文件）
- 详细规范见 `references/xxx.md`
```

**关键原则**：
- **只写「模型不知道的」**：模型会 git add/commit/push，不用写。但「永远不要 force push 到 main」这种陷阱规则必须写。
- **SKILL.md 承载的是路由+核心规则**，不是百科全书。细节放 references/。

---

### Step 4：决定是否需要附属文件

| 文件类型 | 何时需要 | 何时不需要 |
|---------|---------|-----------|
| `scripts/` | 任务有确定性代码逻辑（校验、格式化、模板生成） | 模型可以现场手写且不容易错 |
| `references/` | 有大型规范、Schema、API 文档需要按需查阅 | 内容 < 2000 tokens 且模型需要频繁参考 |
| `assets/` | 需要输出固定模板、图片、配置文件 | 模板可以用文字描述清楚 |
| `examples/` | 任务有歧义，需要示例对齐输出格式 | 输出格式单一、无歧义 |

**核心原则**：能不放就不放。每个附属文件都意味着一笔「运行时税」——Agent 需要主动去读它。

---

### Step 5：验收与测试

**不能只靠手测。** 使用 OpenAI 提出的四维验证体系：

1. **Outcome（结果正确吗？）** → 最低门槛
2. **Process（过程对吗？）** → 备份前不能迁移
3. **Style（风格一致吗？）** → commit message、代码格式
4. **Efficiency（效率够吗？）** → 有没有多余的 API 调用

**一个简单但有效的手测流程**：
1. 用一个干净会话，只加载这个 Skill
2. 输入触发条件的典型用户请求
3. 观察：Agent 是否加载了这个 Skill？过程是否符合预期？
4. 在 SKILL.md 中标记「模型自己就会」的内容 → 删除

---

### Step 6：持续维护

Skill 不是一次性产物。当以下情况发生时，Skill 必须更新：

- 团队的规范变化（如命名规则、代码风格）
- 用户反馈 Agent 在这个任务上持续犯错
- 模型能力升级，某些规则变得冗余

> **Perplexity 的警告**：每一个 Skill 都有一笔「永续税」。写的时候痛快，维护的时候痛苦。控制 Skill 数量，不是越多越好。


## 二、核心写作技巧

### 技巧 1：用「否定式规则」代替「肯定式指南」

| 不推荐（肯定式） | 推荐（否定式） |
|-----------------|---------------|
| 你应该在修改前备份 | 没有备份之前，不要执行修改操作 |
| 建议使用 with 语句打开文件 | 永远不要用 open() 而不加 with |
| 记得检查 API rate limit | 没有检查剩余配额之前，禁止调用 API |

**为什么**：模型在长上下文中的「禁止项」比「应该项」更醒目、更容易被遵循。

---

### 技巧 2：分层披露——SKILL.md 只说「去哪里找」

```markdown
## 代码审查规则
完整清单见 `references/code_review_checklist.md`

关键规则摘要：
- 安全相关规则必须通过，否则直接驳回
- 性能规则允许有例外，但必须备注
```

这样 SKILL.md 保持轻量，Agent 按需加载 checklist 的完整内容。

---

### 技巧 3：用「检查点」替代「长流程描述」

❌ 不要这样写：
```
第一步执行 A，然后检查 A 的输出是否符合条件 X，如果符合则执行 B，否则返回第一步...
```

✅ 这样写：
```
## 检查点 1：A 执行完成
验证条件：输出中包含 "SUCCESS"
通过 → 进入检查点 2
失败 → 返回 Step 1

## 检查点 2：B 开始前
验证条件：文件 /tmp/result.json 已生成
```

检查点让模型容易自检，也方便你在 Eval 中验证「过程是否正确」。

---

### 技巧 4：给模型「对话出口」——Inversion 模式

当信息不足时，不要让模型猜或编。明确告诉模型「应该问用户什么」：

```markdown
## 信息不足时的处理

当用户没有提供以下任一信息时，必须先询问，不得自行假设：
1. 目标环境（staging/production）
2. 是否需要备份
3. 变更影响范围

询问模板：「在执行前，请确认：[问题清单]」
```

---

### 技巧 5：保持「原子性」

一个 Skill 只做一件事。

| 反面案例 | 正面案例 |
|---------|---------|
| `full_stack_development` | `react_component`、`api_design`、`db_migration` |
| `code_review_and_deploy` | `code_review`、`canary_deploy` |

原子性 Skill 可以被组合成复杂工作流，而大而全的 Skill 只会增加「索引税」和认知负担。


## 三、写作规范清单

### 命名规范

| 规则 | 示例 |
|------|------|
| 全小写，只用字母、数字、连字符 | `db-migration` ✅ `DB_Migration` ❌ |
| 目录名与 SKILL.md 中的 name 完全一致 | 目录 `code-review/` → name: `code-review` |
| 名称反映功能，不长于 3 个词 | `pdf-parser` ✅ `a-skill-that-parses-pdf-documents` ❌ |

### Description 规范

| 规则 | 示例 |
|------|------|
| 以 `Load when` 开头（语义上） | `Load when user asks to...` |
| 包含触发关键词 | `mentions "migration", "schema change", "rollback"` |
| 必要时加入负向限定 | `but NOT when user is just asking for documentation` |
| 不超过 200 字符 | — |

### SKILL.md 格式规范

| 规则 | 理由 |
|------|------|
| YAML frontmatter 必须包含 name 和 description | 这是 Agent 的路由依据 |
| 核心规则用无序列表，控制在 10 条以内 | 超过 10 条说明 Skill 不够原子 |
| 工作流程用有序列表 | 顺序敏感 |
| 引用外部文件用 `见 xxx/yyy.md` 格式 | 明确告知 Agent 去读 |
| 不使用复杂格式（表格、代码块嵌套） | 模型解析 markdown 的能力虽好，但越简单越可靠 |

### 附属文件组织规范

```
skill-name/
├── SKILL.md          # 必须存在
├── scripts/          # 可选，放可执行代码
│   └── validate.py
├── references/       # 可选，放按需加载的文档
│   └── api_spec.md
├── assets/           # 可选，放模板/静态资源
│   └── template.html
└── examples/         # 可选，放输入输出示例
    └── sample.md
```

**禁止**：在 references/ 中再嵌套深层目录。Agent 的探索深度有限，最多一层。


## 四、注意事项（坑与避坑）

### ⚠️ 坑 1：SKILL.md 写成了操作手册

**错误示范**：
```
1. 运行 npm install
2. 运行 npm run build
3. 将 dist/ 目录复制到服务器
```

**为什么是坑**：模型本来就会这些基本命令。SKILL.md 被加载一次，就占用一次「加载税」。用宝贵的 Token 位写模型本来就有的知识，是最大的浪费。

**正确做法**：只写「非标准」的部分：
```
## 构建注意事项
- 这个项目使用 PNPM，不是 npm（npm install 会失败）
- 构建前必须执行 `pnpm run type-check`
- 构建产物在 `dist/`，但服务器期望 `build/`，需要额外软链接
```

---

### ⚠️ 坑 2：Description 写成了「功能介绍」

**错误**：`Helps with database migrations and rollbacks`

**为什么是坑**：Agent 看到这个 description，无法判断「什么时候该加载」。Index 税花出去了，但触发条件不明确，Agent 要么永远不加载，要么每次对话都错误加载。

**正确**：`Load when user mentions "migration", "rollback", "schema change", "ALTER TABLE"`

---

### ⚠️ 坑 3：一个 Skill 塞了太多内容

**Perplexity 的教训**：1945 条 IRS 税法塞进一个 Skill，结果模型表现比不用这个 Skill 还差。

**为什么是坑**：当选项数量超过模型的「多选一精度阈值」（通常在几十到一百多），模型开始随机选择或产生幻觉。

**正确做法**：分层——一级目录 10–20 个类别，每个类别下再分 10–15 条细则。用检索工具定位，不要平铺。

---

### ⚠️ 坑 4：忽略了「运行时税」

三类税的总成本：
- **索引税**：每个会话对所有可见 Skill 的 name+description 做一次加载
- **加载税**：SKILL.md 被完整加载时的 Token 消耗
- **运行时税**：Agent 主动读取 references/ 或执行 scripts/ 的成本

很多开发者只算了「加载税」，忽略 Agent 反复读取附属文件产生的「运行时税」。一个简单的原则：**如果 Agent 需要读三次以上的 reference 才能完成任务，说明这个 reference 应该精简或合并进 SKILL.md。**

---

### ⚠️ 坑 5：从未更新过 Skill

Skill 是活的。模型能力在提升，团队规范在变化，用户的失败案例在积累。

**推荐的维护节奏**：
- 每月一次：review 使用频率最低的 20% Skill，考虑删除或合并
- 每次用户反馈 Agent 失败 → 分析是否 Skill 内容问题
- 每次模型大版本升级 → 检查哪些规则已不再需要


## 五、快速自查表（写完后过一遍）

| 检查项 | 通过标准 |
|--------|---------|
| ✅ 原子性 | 一个 Skill 只解决一类任务 |
| ✅ 必要性 | 没有这个 Skill，模型会犯错 |
| ✅ Description 可路由 | 以「Load when」为语义开头，包含触发词 |
| ✅ 只写「模型不知道的」 | SKILL.md 中没有 `git add` 这类基础命令 |
| ✅ 核心规则 ≤ 10 条 | 超过则考虑拆分 |
| ✅ 附属文件有明确加载条件 | SKILL.md 中明确写了「见 xxx」 |
| ✅ 有检查点或显式流程 | 顺序敏感的任务有明确步骤 |
| ✅ 信息不足时有「问」的指令 | 不是猜，不是编 |
| ✅ 通过了四维 Eval | Outcome + Process + Style + Efficiency |
| ✅ 有维护计划 | 知道什么时候该更新/删除 |


> **最后一句话**：写 Skill 不是在写文档，而是在为模型设计「程序性知识的载体」。每一条指令、每一处省略，都在决定 Agent 能不能在真实世界中「干对活」。
