## 1. 核心范式转变

### 1.1 从 Prompt 到 Skill
- **Prompt**: 声明性知识，一次性，散落在对话中，依赖上下文窗口。
- **Skill**: 程序性知识，持久化，文件夹式结构，可复用，自动触发。

### 1.2 程序性知识 (Procedural Knowledge)
真正的智能体不仅需要语言理解，更需要**在真实工作环境中执行任务的能力**。
Skill 是这种能力的载体，它定义了 Agent "怎么做" 而不仅仅是 "知道什么"。

### 1.3 文件夹式技能系统
- **可扩展**: 通过目录结构无限扩展能力。
- **可维护**: 独立文件，版本可控。
- **模块化**: 每个 Skill 是一个独立的能力单元。

## 2. 上下文经济学 (Perplexity 框架)

每一个 Skill 指令都会对 Agent 的上下文窗口征收**永续税**。必须严格评估投入产出比。

### 2.1 三层税制

| 税种 | 定义 | 影响阶段 | 优化策略 |
|:---|:---|:---|:---|
| **索引税 (Index Tax)** | Agent 发现、理解并决定是否激活 Skill 的成本 | 任务分发 | 精准命名，明确触发条件，避免模糊描述 |
| **加载税 (Load Tax)** | 将 Skill 内容加载到上下文窗口的成本 | 初始化 | 精简指令，高密度表达，使用引用代替内联 |
| **运行时税 (Runtime Tax)** | Agent 执行任务时持续携带的上下文成本 | 执行全过程 | 分层加载，按需激活，及时释放，避免冗余 |

**核心定律**: 越靠近上下文入口的指令，税率越高。

### 2.2 必要性测试 (The Mistake Test)
**核心问题**: "没有这条指令，Agent 会犯错吗？"
- **会犯错** → 保留 (P0/P1)
- **不会犯错** → 删除 (上下文税)

### 2.3 Python 之禅的翻转
在 Skill 写作中，**优美、简洁、可读**让位于**有效、明确、可执行**。
- ❌ "代码应该优雅" (无效税)
- ✅ "函数长度 ≤ 30 行" (有效指令)

## 3. Skill 架构与边界

### 3.1 Skill vs Tool vs MCP

| 概念 | 本质 | 示例 |
|:---|:---|:---|
| **Skill** | 程序性知识 + 工作流 + 判断逻辑 | "如何编写 React 组件", "代码审查流程" |
| **Tool** | 原子能力，函数调用 | `read_file`, `run_test`, `search_web` |
| **MCP** | 外部服务协议，标准化接口 | 连接数据库, 调用 API, 访问文件系统 |

### 3.2 何时创建 Skill?
- ✅ 多步骤协调任务
- ✅ 领域特定最佳实践
- ✅ 项目级约定/规范
- ✅ 频繁重复的复杂工作流
- ❌ 单一工具调用可完成
- ❌ 通用常识 (AI 已具备)
- ❌ 一次性任务

## 4. 五大设计模式 (Google ADK)

### 4.1 Router Pattern (路由模式)
- **场景**: 根据输入类型分发到不同处理逻辑。
- **结构**: 识别意图 → 匹配子 Skill → 传递上下文 → 汇总结果。

### 4.2 Orchestrator Pattern (编排模式)
- **场景**: 协调多个子任务完成复杂工作流。
- **结构**: 分解任务 → 按依赖顺序执行 → 检查中间结果 → 输出最终结果。

### 4.3 Code Execution Pattern (代码执行模式)
- **场景**: 需要运行代码并验证结果。
- **结构**: 确认环境 → 执行代码 → 检查输出 → 失败重试/修复 → 清理。

### 4.4 Evaluation Pattern (评估模式)
- **场景**: 判断输出质量或正确性。
- **结构**: 加载标准 → 逐项检查 → 生成报告 → 反馈修改。

### 4.5 Human-in-the-Loop Pattern (人在回路模式)
- **场景**: 涉及不可逆操作或高风险决策。
- **结构**: 识别风险 → 暂停执行 → 展示影响/回滚方案 → 等待确认 → 继续/取消。

## 5. 指令写作法则

### 5.1 指令密度
- **低密度**: 浪费上下文，解释原因。
- **高密度**: 直接给出规则，假设 Agent 理解。

### 5.2 条件指令 (IF-THEN)
- IF [条件] THEN [动作]
- 示例: `IF 组件接收 props THEN 定义 TypeScript 接口`

### 5.3 负面约束
- 明确禁止的操作，防止常见错误。
- 示例: `不要使用 any 类型`, `不要内联样式`

### 5.4 优先级分级
- **P0 (必须)**: 不遵守会出错。
- **P1 (应该)**: 不遵守会降低质量。
- **P2 (建议)**: 不写进 Skill，除非有历史教训。

## 6. 系统化评估 (OpenAI 方法)

### 6.1 评估框架
1. **定义测试用例集**: 正常路径、边界条件、异常路径。
2. **定义评估标准**: 正确性、完整性、一致性。
3. **执行自动化评估**: 运行用例，收集结果，计算通过率。
4. **回归验证**: 修改后重新运行，对比结果，确认无退化。

### 6.2 关键指标
- **通过率**: ≥ 95%
- **一致性**: 多次执行结果一致
- **上下文效率**: 输出质量 / Token 消耗

## 7. Skill 生命周期管理

### 7.1 版本控制
- 使用 `CHANGELOG.md` 记录修改。
- 每次修改必须运行回归测试。

### 7.2 定期审查
- 每月执行一次"必要性测试"。
- 删除过时指令，合并重复内容。

### 7.3 废弃流程
- 标记 `deprecated` → 说明替代方案 → 保留一个周期 → 移除。

## 8. 附录：写作模板与检查清单

### 8.1 SKILL.md 模板

```markdown
# [Skill 名称]

## 触发条件
[精确描述激活条件，降低索引税]

## 概述
[1-2 句话说明目的]

## 核心规则 (P0/P1)
1. [高密度指令]
2. [IF-THEN 条件指令]
3. [负面约束]

## 工作流程
1. [步骤 1]
2. [步骤 2]
3. [步骤 3]

## 参考资源
- [references/ 目录链接]
```

### 8.2 写作检查清单

- [ ] **必要性**: 没有这条指令 Agent 会犯错吗？
- [ ] **密度**: 是否使用了高密度表达？
- [ ] **具体性**: 指令是否具体可执行、可验证？
- [ ] **边界**: 是否混淆了 Skill/Tool/MCP 的职责？
- [ ] **税制**: 是否优化了索引税、加载税、运行时税？
- [ ] **测试**: 是否编写了对应的评估用例？
