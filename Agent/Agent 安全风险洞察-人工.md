# Agent 智能体基础架构与安全分析

> 整理日期：2026/05/13  
> 参考来源：DataWhale Hello-Agents、Claude Code Book 等

---

## 1. 基础知识

智能体（Agent）是基于大语言模型（LLM）的自主执行系统，能够感知环境、进行推理规划、调用外部工具并执行 Actions。与传统 chatbot 不同，Agent 具备**长期记忆**、**工具使用**和**多步骤复杂任务执行**能力。

### 1.1 核心特征

- **自主性**：无需人工干预即可执行多步骤任务
- **工具调用**：通过代码执行、API 调用与外部系统交互
- **记忆与上下文**：基于历史交互进行持续推理
- **目标导向**：根据任务目标动态调整执行路径

随着 Agent 技术在企业级场景（代码生成、数据分析、系统运维）中的深入集成，其安全风险已从理论问题演变为**现实运营威胁**。

---

## 2. 任务环境分析（PEAS 模型）

要理解智能体的运作，必须先理解它所处的**任务环境**。在人工智能领域，通常使用 **PEAS 模型** 来精确描述一个任务环境，即分析其**性能度量（Performance）、环境（Environment）、执行器（Actuators）和传感器（Sensors）**。

### 2.1 环境特性

| 特性 | 说明 | 安全影响 |
|------|------|---------|
| **部分可观察性** | 环境通常是部分可观察的 | 攻击者可隐藏恶意意图 |
| **确定性 vs 随机性** | 行动结果并非总是确定的 | 增加攻击面不确定性 |
| **多智能体环境** | 环境中可能存在其他行动者 | 信任边界模糊、横向移动风险 |
| **序贯性** | 当前动作会影响未来 | 攻击可产生连锁反应 |
| **动态性** | 环境自身可能在智能体决策时发生变化 | 需要快速适应持续变化的威胁 |

## 3. 智能体循环（Agent Loop）

智能体并非一次性完成任务，而是通过一个持续的循环与环境进行交互，这个核心机制被称为 **智能体循环（Agent Loop）**。

### 3.1 循环阶段

```
┌─────────────────────────────────────────────────────────────┐
│                     Agent Loop                              │
│                                                             │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐  │
│  │ 感知    │───▶│ 思考    │───▶│ 行动    │───▶│ 环境    │  │
│  │Percept. │    │ Thought │    │ Action  │    │ Change  │  │
│  └─────────┘    └─────────┘    └─────────┘    └─────────┘  │
│       ▲                                              │     │
│       └────────────────── Observation ◀────────────────┘     │
└─────────────────────────────────────────────────────────────┘
```

| 阶段 | 说明 | 安全风险点 |
|------|------|-----------|
| **感知（Perception）** | 通过传感器接收环境输入（API 端口、用户输入接口） | 恶意输入注入、数据投毒 |
| **思考（Thought）** | LLM 驱动的内部推理过程 | Prompt 注入、推理劫持 |
| ↳ 规划（Planning） | 基于观察和记忆制定行动计划 | 任务分解被操纵 |
| ↳ 工具选择（Tool Selection） | 从工具库中选择适合的工具并确定参数 | 工具滥用、参数注入 |
| **行动（Action）** | 通过执行器执行具体行动（代码解释器、搜索引擎 API） | 沙箱逃逸、RCE |
| **环境变化（State Change）** | 行动引起环境状态变化，产生新的观察反馈 | 状态污染、副作用不可控 |

## 4. 交互协议（Interaction Protocol）

在工程实践中，需要一套明确的**交互协议**来规范 LLM 与环境之间的信息交换。智能体的输出不再是单一的自然语言回复，而是一段遵循特定格式的文本。

### 4.1 协议结构

| 组成部分 | 说明 | 安全关注点 |
|---------|------|-----------|
| **Thought（思考）** | 智能体内部决策的"快照"，阐述如何分析当前情境 | 思考过程可被注入操纵 |
| **Action（行动）** | 基于思考后决定对环境施加的具体操作，通常以函数调用形式表示 | Action 可被劫持执行恶意操作 |
| **Observation（观察）** | 环境返回的处理结果，封装为简洁的自然语言文本 | Observation 可携带恶意指令 |

### 4.2 示例

```
Thought: 用户想知道北京的天气。我需要调用天气查询工具。
Action: get_weather("北京")

Observation: 北京当前天气为晴，气温25摄氏度，微风。
```

**安全洞察**：这个由 Thought → Action → Observation 构成的循环中，每个环节都可能成为攻击面。攻击者可通过污染 Observation 来影响下一轮的 Thought，形成攻击链。

## 5. Claude 架构分析

### 5.1 六层架构

```
┌─────────────────────────────────────────────────────────────┐
│                    可观测性层 (Observability)                │
│              事件总线、审计追踪、生命周期钩子                 │
├─────────────────────────────────────────────────────────────┤
│                    多智能体层 (Multi-Agent)                  │
│           子智能体、智能体团队、Git worktree 隔离             │
├─────────────────────────────────────────────────────────────┤
│                    集成层 (Integration)                      │
│         MCP 运行时、外部服务器连接、跨会话记忆存储            │
├─────────────────────────────────────────────────────────────┤
│                    执行层 (Execution)                        │
│      工具分发、流式运行时、提示词缓存（复用成本约10%）        │
├─────────────────────────────────────────────────────────────┤
│                    知识层 (Knowledge)                        │
│    技能注册表、上下文压缩器（5级联动）、任务图、记忆存储      │
├─────────────────────────────────────────────────────────────┤
│                    输入层 (Input)                            │
│          会话管理、权限控制（permission gating）              │
│                   基于 YAML 的信任分级                        │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 各层安全风险

| 层级 | 核心功能 | 安全风险 |
|------|---------|---------|
| **输入层** | 会话管理、权限控制、信任分级 | 信任分级被绕过、权限控制失效 |
| **知识层** | 技能注册、上下文压缩、任务图、记忆存储 | 上下文压缩触发时的信息丢失、记忆污染 |
| **执行层** | 工具分发、流式运行时、提示词缓存 | 工具调用权限滥用、缓存侧信道泄露 |
| **集成层** | MCP 运行时、外部连接、持久化记忆 | MCP 服务器投毒、跨会话数据泄露 |
| **多智能体层** | 子智能体、智能体团队、worktree 隔离 | 智能体间注入、权限提升链 |
| **可观测性层** | 事件总线、审计追踪、生命周期钩子 | 审计日志被篡改、钩子注入 |

### 5.3 多智能体并行化

| 类型 | 说明 | 安全特性 |
|------|------|---------|
| **子智能体（Subagents）** | 轻量级工作单元，拥有独立上下文窗口，执行集中式任务 | 严格父子层级，无法互相通信，不能生成子智能体 |
| **智能体团队（Agent Teams）** | 团队负责人生成独立团队成员，每个成员作为完整实例运行 | 共享任务列表（JSON）、邮箱系统点对点消息、Git worktree 隔离 |

## 7. Claude 权限设计

### 7.1 权限管线（Permission Pipeline）

权限管线不是简单的"允许/拒绝"开关，而是一套精心设计的多阶段检查机制，在自动化效率与安全控制之间寻找精确的平衡。

**纵深防御思想**：如同现代化办公大楼的门禁系统——大堂只需刷卡，会议室需要预约确认，服务器机房需要指纹验证加双人授权。每一层安全检查都是独立的，即使某一层被绕过，下一层仍然可以阻止未授权的访问。

### 7.2 四阶段管线

```
┌─────────────────────────────────────────────────────────────┐
│                    权限管线流程                              │
│                                                             │
│  工具调用请求                                                │
│       │                                                     │
│       ▼                                                     │
│  ┌─────────────┐                                            │
│  │ 阶段1:      │ ──失败──▶ 拒绝: 输入不合法                 │
│  │ validateInput│                                            │
│  │ (输入验证)  │ ──通过──▶                                  │
│  └─────────────┘           ▼                                │
│                     ┌─────────────┐                         │
│                     │ 阶段2:      │ ──失败──▶ 拒绝: 权限不足 │
│                     │hasPermissions│                         │
│                     │ (规则匹配)  │ ──通过──▶               │
│                     └─────────────┘          ▼              │
│                                      ┌─────────────┐       │
│                                      │ 阶段3:      │       │
│                                      │checkPermiss.│ ──block▶ 拒绝: 钩子拦截 │
│                                      │(上下文评估) │ ──通过──▶              │
│                                      └─────────────┘         ▼              │
│                                                       ┌─────────────┐       │
│                                                       │ 阶段4:      │       │
│                                                       │ canUseTool  │ ──拒绝──▶ 拒绝: 用户拒绝 │
│                                                       │(用户确认)   │ ──允许──▶ 执行工具       │
│                                                       └─────────────┘       │
└─────────────────────────────────────────────────────────────┘
```

| 阶段 | 名称 | 职责 | 短路逻辑 |
|------|------|------|---------|
| **阶段 1** | validateInput | 输入验证，检查数据合法性 | 输入不合法则直接拒绝 |
| **阶段 2** | hasPermissionsToUseTool | 规则匹配，检查权限规则 | 权限不足则直接拒绝 |
| **阶段 3** | checkPermissions | 上下文评估，PreToolUse 钩子前置拦截 | 钩子拦截则拒绝 |
| **阶段 4** | canUseTool | 交互式提示，用户确认 | 用户拒绝则拒绝 |

**设计优势**：早期拒绝（Fail Fast）——如果输入参数不合法（阶段 1），就不需要浪费时间去检查权限规则（阶段 2）；如果权限规则明确拒绝（阶段 2），就不需要弹出用户确认对话框（阶段 4）。

## 8. Claude 工具系统

### 8.1 工具类型契约

每个工具都遵循统一的类型契约：`Tool<Input, Output, Progress>`。通过定义严格的类型接口，工具系统的所有架构约束——权限检查、并发控制、进度报告、UI 渲染——都被编译器强制执行。

### 8.2 核心工具清单

| 类别 | 工具 | 职责 | 并发安全 | 安全风险 |
|------|------|------|---------|---------|
| 执行 | BashTool | 运行 Shell 命令 | 否（副作用） | 命令注入、RCE |
| 文件 | FileReadTool, FileEditTool, FileWriteTool | 读取、编辑、写入文件 | Read 是，Edit/Write 否 | 路径遍历、文件篡改 |
| 搜索 | GlobTool, GrepTool | 文件名模式匹配、内容搜索 | 是 | 信息泄露 |
| 笔记本 | NotebookEditTool | Jupyter Notebook 编辑 | 否 | 代码注入 |
| 网络 | WebFetchTool, WebSearchTool | 获取 URL 内容、网络搜索 | 是 | SSRF、数据外泄 |
| 智能 | AgentTool | 子智能体入口 | 否 | 子智能体劫持 |
| 任务 | TodoWriteTool, TaskCreateTool 等 | 任务管理 | 视具体工具而定 | 任务操纵 |
| 规划 | EnterPlanModeTool, ExitPlanModeV2Tool | 计划模式切换 | 否 | 规划劫持 |
| 交互 | AskUserQuestionTool | 向用户提问 | 否（需要用户响应） | 社会工程 |
| 技能 | SkillTool | 调用 slash command 技能 | 否 | 技能投毒 |
| 配置 | ConfigTool | 修改配置 | 否 | 配置篡改 |
| MCP | ListMcpResourcesTool, ReadMcpResourceTool | MCP 资源访问 | 是 | MCP 投毒 |
| 工作树 | EnterWorktreeTool, ExitWorktreeTool | Git worktree 管理 | 否 | 代码注入 |
| 通知 | BriefTool | 消息发送 | 否 | 信息泄露 |
| 搜索发现 | ToolSearchTool | 延迟工具发现 | 是 | 工具混淆 |

> **设计洞察**：超过一半的工具标记为并发不安全。这反映了一个深刻的工程现实：在 Agent 系统中，大多数操作都有副作用（修改文件、执行命令、更改状态），真正可以安全并行执行的操作（纯读取、纯搜索）是少数。

## 9. Claude 技能与插件

### 9.1 技能系统架构

技能系统是一个多层次的扩展机制，允许用户通过 Markdown 文件定义可复用的 prompt 模板，也允许开发者通过 TypeScript 代码注册编译时内置的技能。设计目标：**零配置可用，有配置强大**。

### 9.2 内置技能分类

| 类型 | 说明 | 注册方式 |
|------|------|---------|
| **核心技能** | 始终注册 | 启动时通过 `initBundledSkills()` 统一注册 |
| **Feature-gated 技能** | 通过功能门控，仅在对应功能标志启用时注册 | 功能标志控制 |

### 9.3 核心内置技能清单

| 技能名称               | 用途                     | 用户可调用 | 典型使用场景             | 安全风险 |
| ------------------ | ---------------------- | ----- | ------------------ | ---- |
| `update-config`    | 配置 settings.json 的各项设置 | 是     | 修改权限规则、环境变量、默认模型   | 配置篡改 |
| `keybindings-help` | 键盘快捷键自定义帮助             | 是     | 查看和自定义按键绑定         | 低    |
| `verify`           | 验证代码变更的正确性             | 是     | 提交前的最终验证、CI 前的本地检查 | 验证绕过 |

| `skillify`         | 将 prompt 转换为可复用的技能     | 是     | 将一次性的 prompt 模板化   | 技能投毒 |
| `remember`         | 记忆管理（CLAUDE.md 条目）     | 是     | 添加项目规范、团队约定        | 记忆污染 |
| `batch`            | 批量文件处理                 | 是     | 批量重命名、批量格式化        | 批量破坏 |

| `lorem-ipsum`      | 生成占位内容                 | 是     | UI 开发时的占位文本        | 低    |

## 10. Claude MCP 集成

### 10.1 MCP 设计原则

| 原则 | 说明 | 安全意义 |
|------|------|---------|
| **协议即契约** | MCP 定义了客户端与服务器之间的严格消息格式和能力声明机制 | 标准化接口减少攻击面 |
| **传输无关性** | MCP 协议本身不绑定任何特定的网络传输方式 | 支持多种部署场景 |
| **安全边界内嵌** | 将安全策略集成到协议的每一个层次 | 默认不信任，即使接入不可信第三方也不会危及系统 |

### 10.2 传输协议对比

| 协议类型 | 传输方式 | 延迟特征 | 适用场景 | 安全风险 |
|---------|---------|---------|---------|---------|
| `stdio` | 标准输入/输出管道 | 最低（进程内通信） | 本地开发工具、文件系统操作 | 进程注入 |
| `sse` | Server-Sent Events | 网络延迟 | 远程 HTTP 服务、云端部署 | 中间人攻击 |
| `sse-ide` | SSE + IDE 元数据 | 本地网络 | IDE 扩展专用 | IDE 元数据泄露 |
| `http` | HTTP Streamable | 网络延迟 | MCP 规范新协议，支持流式响应 | HTTP 注入 |
| `ws` | WebSocket 全双工 | 网络延迟 | 需要实时双向通信的场景 | WebSocket 劫持 |
| `ws-ide` | WebSocket + IDE 元数据 | 本地网络 | IDE 扩展专用，需要低延迟双向通信 | IDE 元数据泄露 |
| `sdk` | 进程内函数调用 | 近零延迟 | SDK 内部调用，不启动实际进程或网络连接 | SDK 注入 |
| `claudeai-proxy` | Claude.ai 代理 | 网络延迟 | Claude.ai 平台代理服务器 | 代理劫持 |

## 11. Agent Harness 设计原则

### 11.1 五大设计原则

| 原则 | 说明 | 安全意义 |
|------|------|---------|
| **循环优于递归** | 核心查询函数是 `AsyncGenerator`，通过 `while(true)` 循环管理状态流转 | 状态恢复更自然、中止更可控、调试更直观 |
| **Schema 驱动而非硬编码** | 每个工具都通过 Zod Schema 定义输入参数 | 单一真相源，消除不一致的根源 |
| **渐进式权限** | 权限系统分为四个阶段，每个阶段都可以短路返回 | 早期拒绝（Fail Fast），减少攻击面 |
| **流式优先** | 所有数据都通过 `AsyncGenerator` 的 `yield` 传递 | 实时处理，减少等待时间 |
| **可插拔扩展** | 钩子系统在二十多个生命周期节点提供扩展点 | 标准化扩展机制，便于安全审计 |

### 11.2 循环优于递归的安全优势

| 优势 | 循环模式 | 递归模式 |
|------|---------|---------|
| **状态恢复** | 只需重新赋值 `state` 变量 | 需要回退整个调用栈 |
| **中止控制** | `AbortController` 信号检查在循环顶部是自然退出点 | 中止信号需要在每一层递归中传递和检查 |
| **调试直观性** | 状态变化发生在固定代码位置，一个断点捕获所有转换 | 状态变化分散在多个调用栈帧中 |

## 12. 安全风险评估

基于上述架构分析，识别以下关键安全风险：

### 12.1 架构级风险

| 风险类型 | 影响层级 | 严重程度 | 描述 |
|---------|---------|---------|------|
| **权限管线绕过** | 输入层、执行层 | ⭐⭐⭐⭐⭐ | 攻击者可能绕过四阶段权限检查 |
| **上下文压缩信息泄露** | 知识层 | ⭐⭐⭐⭐ | 上下文压缩时可能泄露敏感信息 |
| **缓存侧信道攻击** | 执行层 | ⭐⭐⭐⭐ | 提示词缓存可能残留敏感数据 |
| **MCP 服务器投毒** | 集成层 | ⭐⭐⭐⭐⭐ | 恶意 MCP 服务器可提供被污染的工具 |
| **智能体间注入** | 多智能体层 | ⭐⭐⭐⭐⭐ | 一个 Agent 的输出被用作另一 Agent 的输入 |
| **审计日志篡改** | 可观测性层 | ⭐⭐⭐⭐ | 攻击者可能篡改审计追踪 |

### 12.2 工具级风险

| 风险类型 | 影响工具 | 严重程度 | 描述 |
|---------|---------|---------|------|
| **命令注入** | BashTool | ⭐⭐⭐⭐⭐ | 通过 Shell 命令执行任意代码 |
| **路径遍历** | FileReadTool, FileEditTool, FileWriteTool | ⭐⭐⭐⭐ | 访问或修改非授权路径的文件 |
| **SSRF** | WebFetchTool, WebSearchTool | ⭐⭐⭐⭐ | 通过工具访问内部网络资源 |
| **子智能体劫持** | AgentTool | ⭐⭐⭐⭐ | 控制子智能体执行恶意操作 |
| **技能投毒** | SkillTool | ⭐⭐⭐⭐ | 恶意技能定义影响 Agent 行为 |

### 12.3 协议级风险

| 风险类型 | 影响协议 | 严重程度 | 描述 |
|---------|---------|---------|------|
| **中间人攻击** | sse, http, ws | ⭐⭐⭐⭐ | 拦截和篡改传输中的数据 |
| **进程注入** | stdio, sdk | ⭐⭐⭐⭐ | 通过进程通信注入恶意代码 |
| **代理劫持** | claudeai-proxy | ⭐⭐⭐⭐⭐ | 控制代理服务器窃取数据 |
